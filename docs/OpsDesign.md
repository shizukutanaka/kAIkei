# kAIkei — 運用設計書

> 本ドキュメントはkAIkeiのSLA・障害対応・変更管理・バックアップ運用手順を定義する。
> 運用チーム・オンコール担当者の基準資料である。

---

## 1. サービスレベルアグリーメント（SLA）

### 1.1 可用性目標

| 項目 | 目標値 | 測定方法 |
|------|--------|---------|
| 稼働率 | 99.9%以上/月 | `(総時間 - ダウンタイム) / 総時間` |
| 計画停止 | 月1回・4時間以内 | 事前告知のメンテナンス時間 |
| 計画外停止 | 月間合計44分以内 | 99.9%の計算: `43200分 × 0.001 = 43.2分` |

### 1.2 性能目標

| 項目 | 目標値 | 測定方法 |
|------|--------|---------|
| API応答時間（p95） | < 500ms | APM（Datadog/New Relic） |
| API応答時間（p99） | < 2秒 | 同上 |
| 画面ロード時間（TTI） | < 3秒 | Lighthouse / RUM |
| バッチ処理（月次クローズ） | < 10分 | バッチジョブログ |
| AI推論応答時間 | < 5秒（p95） | APIログ |

### 1.3 目標復旧時間

| 項目 | 目標値 | 備考 |
|------|--------|------|
| RPO（Recovery Point Objective） | 5分以内 | WALアーカイブ間隔 |
| RTO（Recovery Time Objective） | 3時間以内 | フル復旧までの目標時間 |
| 障害検知時間 | 5分以内 | 監視アラート通知まで |
| 障害対応開始時間 | 15分以内 | オンコール担当者の確認まで |

## 3. 障害対応

### 3.1 障害対応フロー

```
[アラート検知]
  → [レベル判定]
    → P1: 即時エスカレーション
      → オンコール担当者が確認（15分以内）
        → 影響範囲特定
          → 緊急対応（ロールバック/切り替え/再起動）
            → 復旧確認
              → インシデントレポート作成（24時間以内）
                → 根本原因分析（RCA）（5営業日以内）
                  → 再発防止策実施
    → P2: 営業時間内優先対応
      → 影響範囲特定
        → 修正対応 or ワークアラウンド
          → 復旧確認
            → インシデントレポート作成（48時間以内）
    → P3: 営業時間内対応
      → バックログ登録 or 即時対応
    → P4: 記録のみ
      → 定期レビューで確認
```

### 3.2 インシデントレポートフォーマット

```markdown
## インシデントレポート: INC-YYYYMMDD-NNN

| 項目 | 内容 |
|------|------|
| 発生日時 | YYYY-MM-DD HH:MM JST |
| 検知日時 | YYYY-MM-DD HH:MM JST |
| 復旧日時 | YYYY-MM-DD HH:MM JST |
| 影響時間 | XX分 |
| レベル | P1 / P2 / P3 |
| 影響範囲 | テナントX・会社Y・全ユーザー 等 |
| 影響ユーザー数 | XXX人 |
| 現象 | （障害の概要） |
| 原因 | （根本原因） |
| 対応内容 | （実施した復旧作業） |
| 再発防止策 | （今後の対策） |
| 教訓 | （得られた知見） |
```

### 3.3 エスカレーションパス

| レベル | 担当 | エスカレーション先 |
|--------|------|-------------------|
| 第1階層 | オンコール担当者 | — |
| 第2階層 | 運用リード | 第1階層が15分以内に対応できない場合 |
| 第3階層 | CTO / 技術責任者 | P1が30分以内に復旧しない場合 |
| 第4階層 | CEO / 経営層 | P1が1時間以内に復旧しない場合 |

## 4. バックアップ運用

### 4.1 バックアップスケジュール

| 種別 | 頻度 | 実行時刻 | 保持期間 | 保存先 |
|------|------|---------|---------|--------|
| フルバックアップ | 毎日 | 02:00 JST | 30日 | 別リージョンS3 |
| 増分バックアップ | 毎時 | 毎正時 | 7日 | 別リージョンS3 |
| WALアーカイブ | リアルタイム | トランザクション毎 | 90日 | 別リージョンS3 |
| 月次フルバックアップ | 毎月1日 | 02:00 JST | 12ヶ月 | 別リージョンS3（Glacier） |
| 年次フルバックアップ | 毎年1月1日 | 02:00 JST | 7年 | 別リージョンS3（Glacier Deep Archive） |

### 4.2 バックアップ検証

| 項目 | 頻度 | 手順 | 合格基準 |
|------|------|------|---------|
| バックアップ完了確認 | 毎日 | バックアップジョブログ確認 | ステータス=success |
| リストアテスト | 月1回 | ステージング環境へリストア | データ整合性OK |
| PITRテスト | 四半期1回 | 指定時点への復元実行 | 復元データが正確 |
| リージョンフェイルオーバー | 半年1回 | ディザスタリカバリ訓練 | RTO 3時間以内に復旧 |

### 4.3 バックアップ失敗時の対応

```
[バックアップジョブ失敗検知]
  → アラート通知（P2）
    → [原因特定]
      → ディスク容量不足 → 不要ファイル削除 → 再実行
      → ネットワーク障害 → 接続復旧後に再実行
      → DBエラー → DBログ確認 → 復旧後に再実行
        → [3回再試行しても失敗?]
          ─Yes→ P1にエスカレーション → 手動バックアップ検討
          ─No→ 完了確認 → 終了
```

## 5. ディザスタリカバリ（DR）

### 5.1 DR構成

```
[プライマリリージョン: 東京]
  ├── Application Server (FastAPI)
  ├── PostgreSQL (Primary)
  ├── Redis (Cache/Queue)
  └── S3 (Documents)

  ↓ (非同期レプリケーション)

[DRリージョン: 大阪]
  ├── Application Server (Standby - 停止状態)
  ├── PostgreSQL (Replica - 読み取り専用)
  ├── Redis (Standby)
  └── S3 (Cross-Region Replication)
```

### 5.2 フェイルオーバー手順

```
[1] 障害判定（プライマリリージョン復旧不可と判断）
  → 管理者の承認

[2] DRリージョンのPostgreSQLを昇格
  → `pg_promote()` 実行（Replica → Primary）

[3] DNS切り替え
  → Route53 / Cloudflare でDRリージョンへ向ける
  → TTL 60秒以下に設定済み

[4] Application Server起動
  → DRリージョンのApp Serverを起動
  → 環境変数・DB接続先をDRリージョンに切り替え

[5] Redis起動
  → DRリージョンのRedisを起動

[6] サービス復旧確認
  → ヘルスチェックエンドポイント確認
  → スモークテスト実行

[7] ユーザー通知
  → メンテナンスページ解除
  → 障害通知送信

[目標: 全工程3時間以内に完了]
```

### 5.3 DR訓練

| 項目 | 頻度 | 内容 |
|------|------|------|
| テーブルトップ訓練 | 四半期1回 | フェイルオーバー手順の机上確認 |
| 実機フェイルオーバー訓練 | 半年1回 | 実際にDRリージョンへ切り替え・復旧確認 |
| データ整合性検証 | 半年1回 | DRリージョンのデータがプライマリと一致することを確認 |

## 6. 変更管理

### 6.1 変更区分

| 区分 | 定義 | 承認者 | リードタイム |
|------|------|--------|------------|
| 通常変更 | 機能追加・バグ修正・設定変更 | 運用リード | 3営業日 |
| 緊急変更 | 本番障害の緊急修正 | 運用リード（事後承認可） | 即時 |
| 標準変更 | 事前承認済みの反復作業（定常バッチ等） | 変更管理委員会（初回のみ） | 事前承認済み |

### 6.2 変更プロセス

```
[1] 変更要求の起票
  → Change Request（CR）作成
    → 変更内容・影響範囲・リスク評価・ロールバック手順を記載

[2] レビュー・承認
  → 運用リードがCRをレビュー
    → [承認?]
      ─No→ 差戻し → 修正 → 再レビュー
      ─Yes→ スケジュール確定

[3] ステージング環境での検証
  → デプロイ → 自動テスト実行 → 結果確認

[4] 本番デプロイ
  → Feature Flagを用いた段階的ロールアウト
    → 5% → 25% → 50% → 100%
      → [各段階で障害?]
        ─Yes→ 自動ロールバック → 原因分析
        ─No→ 次段階へ

[5] 変更完了確認
  → 本番環境での動作確認
  → 変更記録のクローズ
```

### 6.3 ロールバック手順

| 変更種別 | ロールバック方法 | 所要時間 |
|---------|----------------|---------|
| アプリケーション | 前バージョンのコンテナイメージに切り替し | 5分 |
| DBスキーマ変更 | Alembicダウンマイグレーション | 10分 |
| 設定変更 | 前バージョンの設定ファイルに復元 | 5分 |
| Feature Flag | FlagをOFFに切り替え | 即時 |
| DBデータ修正 | PITRで変更前時点へ復元 | 30分〜3時間 |
