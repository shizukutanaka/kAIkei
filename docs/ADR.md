# kAIkei — アーキテクチャ決定記録（ADR）

> 本ドキュメントはkAIkeiの技術選定における決定根拠・代替案・制約を記録する。
> 各ADR（Architecture Decision Record）は番号付きで管理し、将来の見直し時に判断材料を提供する。

## ADR-001: データベースにPostgreSQLを採用する

| 項目 | 内容 |
|------|------|
| ステータス | 採用済み |
| 決定日 | 2026-06-01 |

### 背景

kAIkeiは多テナント・多会社・履歴保持（Event Sourcing / Valid From-To）を要件とする。
金融データを扱うため、ACIDトランザクション・JSONBサポート・RLS（Row Level Security）・PITRが必須。

### 決定

PostgreSQL 16以上を採用する。

### 根拠

- **ACID準拠**: 会計データの一貫性が不可欠。PostgreSQLは強固なACID保証を提供。
- **RLS（Row Level Security）**: マルチテナント分離をDB層で実現可能。アプリケーション層の漏れを防ぐ。
- **JSONB**: AI推論結果・設定データなど非構造化データの格納に柔軟に対応。
- **PITR（Point-in-Time Recovery）**: WALアーカイブにより任意時点への復元が可能。RPO 5分を達成。
- **UUID主キー**: `gen_random_uuid()`でアプリ層不要のUUID生成が可能。
- **パーティショニング**: 大量仕訳データの月次パーティショニングでクエリ性能を確保。

### 代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| MySQL 8.x | RLSがネイティブでサポートされていない。JSON型の機能がPostgreSQLに比べて劣る。 |
| Oracle Database | ライセンス費用が高額。オープンソース要件に合わない。 |
| MongoDB | ACIDトランザクションの保証が弱い（ドキュメント間トランザクションは制限あり）。会計データに不適。 |

### 制約・影響

- 運用チームにPostgreSQLの専門知識が必要。
- バックアップ・リカバリ手順の整備が必須（WALアーカイブ設定）。
- 接続プール管理にPgBouncer等の導入を推奨。

## ADR-002: プライマリキーにUUIDを採用する

| 項目 | 内容 |
|------|------|
| ステータス | 採用済み |
| 決定日 | 2026-06-01 |

### 背景

マルチテナント・マルチ会社環境で、データのインポート・移行時に主キーの衝突を避ける必要がある。
また、分散環境でのID生成・将来のシャーディングも考慮する。

### 決定

全テーブルの主キーにUUID v4（`gen_random_uuid()`）を採用する。

### 根拠

- **衝突回避**: 移行元ソフトからのデータインポート時、ID衝突のリスクを排除。
- **分散生成**: アプリケーション層でIDを事前生成可能（クライアント側でIDを確定してからPOST）。
- **将来のシャーディング**: UUIDは自然に分散可能なため、水平スケール時の再設計が不要。
- **セキュリティ**: 連番ID（1, 2, 3...）と異なり、レコード数の推測を防止。

### 代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| SERIAL（自動連番） | 移行時のID衝突リスク。レコード数の推測が可能。 |
| BIGSERIAL + テナントプレフィックス | 複雑性が増す。分散環境での一意性保証が困難。 |
| ULID | 時系列ソート可能だが、PostgreSQLネイティブサポートがない。 |

### 制約・影響

- UUIDは16バイト（128ビット）のため、インデックスサイズがBIGINT（8バイト）の2倍。
- 時系列順序が保証されないため、ソートには `created_at` カラムを併用する必要がある。
- インデックス性能への影響を緩和するため、BRINインデックスの併用を検討。

## ADR-003: フロントエンドにReact + TailwindCSS + shadcn/uiを採用する

| 項目 | 内容 |
|------|------|
| ステータス | 採用済み |
| 決定日 | 2026-06-01 |

### 背景

ERPシステムは多数のフォーム・テーブル・モーダルを持ち、一貫したUI/UXと高い開発効率が求められる。
また、WCAG 2.1 AA準拠のアクセシビリティ要件がある。

### 決定

React 18+ / Next.js（App Router）/ TailwindCSS / shadcn/ui を採用する。

### 根拠

- **React**: エコシステムが最大規模。コンポーネント再利用性が高い。
- **Next.js**: SSR/SSG/ISRの柔軟な切り替え。API RoutesでBFF層を構築可能。
- **TailwindCSS**: ユーティリティファーストでカスタマイズ性が高い。デザインシステムの構築が容易。
- **shadcn/ui**: Radix UIベースのアクセシブルなコンポーネント。コピー&ペースト方式でカスタマイズ自由。
- **React Hook Form + Zod**: フォームバリデーションの型安全性とパフォーマンスを両立。

### 代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| Vue 3 + Vuetify | エコシステムがReactに比べて小さい。shadcn/ui相当のライブラリが少ない。 |
| Angular Material | 学習コストが高い。バンドルサイズが大きい。 |
| SvelteKit | エコシステムがまだ小さい。ERP規模のプロジェクト実績が少ない。 |

### 制約・影響

- 開発チームにReact/TypeScriptの知識が必要。
- shadcn/uiはコピー&ペースト方式のため、コンポーネントのアップストリーム追従が手動。

## ADR-004: AI推論エンジンにOpenAI GPT-4o / Claudeをデュアル採用する

| 項目 | 内容 |
|------|------|
| ステータス | 採用済み |
| 決定日 | 2026-06-01 |

### 背景

仕訳推論・税務予測・監査異常検知など、複数のAIユースケースがある。
単一ベンダーロックインを避け、推論品質の比較・フォールバックを可能にする必要がある。

### 決定

OpenAI GPT-4o と Anthropic Claude をデュアルプロバイダーとして採用し、抽象レイヤーで切り替え可能にする。

### 根拠

- **ベンダーロックイン回避**: 両プロバイダーのAPI仕様を抽象化し、切り替え・フォールバックが可能。
- **推論品質比較**: 同一入力に対する両モデルの出力を比較し、より精度の高い結果を採用可能。
- **可用性**: 一方のAPI障害時に他方へフォールバック。
- **コスト最適化**: ユースケースに応じてコスト性能比の良いモデルを選択可能。

### 代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| OpenAI単体 | ベンダーロックインリスク。API障害時のフォールバックがない。 |
| オープンソースLLM（Llama等） | セルフホストのコスト・運用負荷が高い。推論品質が商用モデルに劣る可能性。 |
| Azure OpenAI Service | OpenAIと同等だが、Azureインフラへの依存が生じる。 |

### 制約・影響

- APIキー管理・レート制限・コスト監視の仕組みが必要。
- プロンプトエンジニアリングがモデルごとに異なる可能性がある。
- データプライバシー: 機密データの送信前にマスキング処理が必須。

## ADR-005: API認証にJWT + Refresh Tokenを採用する

| 項目 | 内容 |
|------|------|
| ステータス | 採用済み |
| 決定日 | 2026-06-01 |

### 背景

マルチテナント環境で、ステートレスな認証が必要。SSO（SAML/OIDC）との統合も要件。

### 決定

- アクセストークン: JWT（有効期限15分）
- リフレッシュトークン: HTTP-only Cookie（有効期限7日）
- SSO統合: SAML 2.0 / OIDCからJWTへの変換レイヤーを設ける

### 根拠

- **ステートレス**: JWTはサーバー側でセッション状態を保持不要。水平スケールが容易。
- **RLS連携**: JWTのclaimに `tenant_id` / `company_id` / `role` を含め、PostgreSQL RLSのポリシーに渡す。
- **SSO統合**: SAML/OIDCの認証結果をJWTに変換することで、アプリ層はJWTのみを扱えばよい。
- **セキュリティ**: 短命アクセストークン + HTTP-only Cookieのリフレッシュトークンで、XSS/CSRFリスクを軽減。

### 代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| セッションベース認証 | ステートフルなため水平スケールが困難。 |
| OAuth 2.0のみ | 内部認証にOAuthは過剰。外部API連携時のみOAuthを使用。 |
| API Keyのみ | ユーザー単位の認可が困難。ロールベースアクセス制御に不適。 |

### 制約・影響

- JWTの失効（リボーク）が困難なため、ブラックリスト機構または短命トークンで対応。
- リフレッシュトークンのローテーション（使用ごとに新トークン発行）で盗難対策。

## ADR-006: API冪等性をIdempotency-Keyヘッダーで実装する

| 項目 | 内容 |
|------|------|
| ステータス | 採用済み |
| 決定日 | 2026-06-01 |

### 背景

クライアントのネットワークエラーによるリトライで、同じ仕訳が二重作成されることを防ぐ必要がある。

### 決定

全POSTリクエストで `Idempotency-Key` ヘッダーをサポートし、`idempotency_records` テーブルで重複を検出する。

### 根拠

- **二重作成防止**: 同一キーでの再リクエストは最初のレスポンスを返却（201/200）。
- **標準的アプローチ**: Stripe等の主要APIで実績のあるパターン。
- **クライアント負荷軽減**: リトライ時の複雑な状態確認が不要。

### 代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| クライアント側で一意IDを生成してPOST bodyに含める | ヘッダーベースの方がRESTfulでクリーン。 |
| サーバー側で重複検知（内容ベース） | 内容が同一でも意図的な重複（定期仕訳等）と区別できない。 |

### 制約・影響

- `idempotency_records` テーブルのTTL管理（24時間でパージ）。
- 異なるリクエストボディで同一キーを使用した場合は409 Conflictを返す。

## ADR-007: 履歴管理にEvent Sourcingパターンを採用する（一部テーブル）

| 項目 | 内容 |
|------|------|
| ステータス | 採用済み |
| 決定日 | 2026-06-01 |

### 背景

勘定科目マスタ・承認ポリシー・税ルールなど、変更履歴の追跡が必要なエンティティがある。
過去時点の状態を再現できる必要がある（例: 過去年度の税計算時に当時の税率を適用）。

### 決定

- マスタデータ: Valid From-To（期間管理）方式を採用。
- 監査ログ: `audit_trails` テーブルに全操作を記録（Event Sourcing的アプローチ）。
- 仕訳データ: 変更不可（immutable）。無効化（void）のみ可能。

### 根拠

- **Valid From-To**: マスタの変更時、既存レコードの `valid_to` を設定し、新レコードを `valid_from` で挿入。
- **過去再現**: 任意時点でのマスタ状態をクエリ可能（`valid_from <= date AND valid_to > date`）。
- **仕訳の不変性**: 会計データの信頼性確保。修正は無効化＋新規作成で対応。

### 代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| 全テーブルEvent Sourcing | シンプルなCRUDテーブルには過剰。実装コストが高い。 |
| 変更履歴テーブル（audit logのみ） | 過去時点のマスタ状態のクエリが複雑になる。 |
| CQRS完全分離 | 読み取り・書き込みモデルの分離はERP規模では過剰。 |

### 制約・影響

- Valid From-To管理テーブルのクエリが複雑になる（期間重複チェック等）。
- マスタ更新時に既存レコードの `valid_to` 更新と新レコード挿入をトランザクション内で実行。

## ADR-008: バックエンドにPython（FastAPI）を採用する

| 項目 | 内容 |
|------|------|
| ステータス | 採用済み |
| 決定日 | 2026-06-01 |

### 背景

会計エンジン・AI推論連携・税計算ロジックなど、数値計算・AI統合が中心のバックエンドが必要。

### 決定

Python 3.12+ / FastAPI をバックエンドフレームワークとして採用する。

### 根拠

- **AI統合**: OpenAI/Anthropic SDKがPythonファースト。AI推論パイプラインの実装が容易。
- **数値計算**: Pythonの `decimal` モジュールで高精度な金額計算が可能。
- **FastAPI**: Pydanticベースの型安全なリクエスト/レスポンス検証。OpenAPIスキーマの自動生成。
- **非同期サポート**: async/awaitでI/O集約的な処理（DB・外部API呼び出し）を効率化。
- **エコシステム**: SQLAlchemy（ORM）・Alembic（マイグレーション）・pytest（テスト）の成熟したエコシステム。

### 代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| Node.js (Express/NestJS) | AI SDKのPython版がより充実。数値計算の精度管理がPythonより煩雑。 |
| Go | AI統合ライブラリが少ない。ビジネスロジックの記述が冗長になる傾向。 |
| Java (Spring Boot) | 開発速度がPythonに比べて遅い。インフラリソース消費が大きい。 |

### 制約・影響

- PythonのGIL制約があるため、CPU集約的な処理はマルチプロセスで対応。
- 型ヒント（mypy）の徹底で実行時エラーを防止。
- 本番環境ではuvicorn + gunicorn（ワーカー複数）で運用。

## ADR-009: オブジェクトストレージにS3互換ストレージを採用する

| 項目 | 内容 |
|------|------|
| ステータス | 採用済み |
| 決定日 | 2026-06-01 |

### 背景

証憑ファイル（PDF/JPEG/PNG）の保存に、スケーラブルで耐久性の高いストレージが必要。
電帳法要件により、ファイルの改ざん検知・長期保存が必須。

### 決定

S3互換オブジェクトストレージ（AWS S3 / MinIO / Cloudflare R2 等）を採用する。

### 根拠

- **耐久性**: S3の11ナイン（99.999999999%）の耐久性。
- **スケーラビリティ**: 容量制限なし。ファイル数増加に自動対応。
- **コスト**: ブロックストレージに比べて単位GBあたりのコストが低い。
- **S3互換**: MinIO（オンプレ）・R2（CDN統合）等への移行が容易。ベンダーロックイン回避。
- **バージョニング**: S3バージョニングでファイルの変更履歴を保持（電帳法要件）。

### 代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| PostgreSQLのBYTEAカラム | DBサイズが肥大化。バックアップ・リストア時間が増大。 |
| ファイルシステム（NFS等） | スケーラビリティ・耐久性が不十分。分散環境での一貫性管理が複雑。 |
| Azure Blob Storage | S3互換ではないため、ベンダーロックインリスク。 |

### 制約・影響

- ファイルアクセスはPresigned URL経由で行う（アプリ層を経由しない直接DL）。
- ライフサイクルポリシーで長期保存ファイルをGlacier等の低コスト層へ移行。

## ADR-010: ワークフローエンジンにカスタム軽量エンジンを採用する

| 項目 | 内容 |
|------|------|
| ステータス | 採用済み |
| 決定日 | 2026-06-01 |

### 背景

仕訳承認・支払承認・経費精算など、多段階承認ワークフローが必要。
BPMN定義はドキュメント化するが、実行エンジンは軽量な独自実装で十分。

### 決定

FastAPI上で動作するカスタム軽量ワークフローエンジンを採用する。
承認ステップ・条件分岐・金額段階制御を `approval_policies` テーブルでドリブンする。

### 根拠

- **要件カバレッジ**: 承認ワークフローの要件（直列・並列・金額段階・SoDチェック）は独自実装で十分。
- **軽量性**: Camunda等のフルBPMNエンジンは機能過剰。運用コストが高い。
- **DBドリブン**: 承認ポリシーをDBマスタ化し、コード変更なしでワークフロー変更が可能。
- **デバッグ性**: 独自実装のため、ログ・デバッグが容易。

### 代替案

| 代替案 | 不採用理由 |
|--------|-----------|
| Camunda Platform | フルBPMNエンジンは機能過剰。ライセンス費用・運用負荷が高い。 |
| Temporal | ワークフローエンジンとしては強力だが、承認フローには過剰。学習コストが高い。 |
| Zapier / n8n | 外部SaaS依存。ERPの内部ワークフローには不適。 |

### 制約・影響

- 複雑なワークフロー（ループ・タイマー境界イベント等）が必要になった場合は再検討。
- BPMN仕様書（`BPMNspec.md`）はドキュメントとして維持し、将来のエンジン移行に備える。
