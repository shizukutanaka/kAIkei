# kAIkei — データ移行仕様書

> 本ドキュメントは既存会計ソフトからkAIkeiへのデータ移行手順・検証方法・ロールバック手順を定義する。

---

## 1. 移行対象ソフトウェア

| ソフト | 移行方式 | エクスポート形式 | 備考 |
|--------|---------|----------------|------|
| 弥生会計 | 専用アダプタ | CSV（弥生標準形式） | 勘定科目・補助科目・仕訳データ |
| freee会計 | API / CSV | CSV（freeeエクスポート） | API経由で全データ取得可能 |
| マネーフォワード クラウド会計 | CSV | CSV（MF標準形式） | 管理画面からエクスポート |
| Dreamvison | CSV | CSV（独自形式） | カスタムマッピング定義 |
| 汎用CSV | 汎用アダプタ | CSV（汎用フォーマット） | ユーザーがカラムマッピングを定義 |

## 2. 移行データ範囲

### 2.1 マスタデータ

| データ | 移行元 | 移行先テーブル | 必須/任意 | 備考 |
|--------|--------|--------------|----------|------|
| 勘定科目 | 科目マスタ | `accounts` | 必須 | 科目コード・名称・貸借区分 |
| 補助科目 | 補助科目マスタ | `sub_accounts` | 任意 | 親科目に紐づく |
| 部門 | 部門マスタ | `departments` | 任意 | 階層構造は平坦化して移行 |
| 取引先 | 取引先マスタ | `partners` | 任意 | 口座情報含む |
| 税区分 | 税区分マスタ | `tax_rules` | 必須 | kAIkei標準にマッピング |
| 社員 | 社員マスタ | `employees` | 任意 | 給与モジュール使用時 |

### 2.2 トランザクションデータ

| データ | 移行先テーブル | 期間 | 備考 |
|--------|--------------|------|------|
| 仕訳（開始残高） | `journal_headers` + `journal_lines` | 移行基準日の期首残高 | 借貸一致必須 |
| 仕訳（過去分） | `journal_headers` + `journal_lines` | 当期1月1日〜移行基準日 | 全仕訳を移行 |
| 証憑ファイル | `archived_documents` | 当期分 | ファイル本体も移行 |
| 固定資産 | `fixed_assets` | 全有形固定資産 | 取得価額・累計償却額含む |
| 未決済支払 | `payment_requests` | 移行基準日時点 | status=pendingで移行 |

### 2.3 移行対象外データ

| データ | 理由 | 代替手段 |
|--------|------|---------|
| 過年度の月次残高 | 仕訳データから再計算可能 | 移行後にバッチで再計算 |
| 監査ログ | 移行元の操作履歴は不要 | kAIkeiの監査ログは移行日から開始 |
| AI推論ログ | 移行元にAI機能なし | — |
| Webhook配信履歴 | 移行元にWebhook機能なし | — |

## 3. 移行プロセス

### 3.1 全体フロー

```
[Phase 1: 事前準備]
  → 移行元ソフトからデータエクスポート
  → フォーマット変換（アダプタ実行）
  → カラムマッピング定義

[Phase 2: Dry-runシミュレーション]
  → テスト環境へインポート
  → バリデーションエラー確認
  → 差分レポート生成
  → [エラーあり?]
    ─Yes→ マッピング修正 → 再シミュレーション
    ─No→ Phase 3へ

[Phase 3: 本番移行]
  → 移行基準日確定（月末推奨）
  → 移行元の新規入力停止（フリーズ）
  → 本番環境へインポート実行
  → 進捗モニタリング

[Phase 4: 検証]
  → 残高検証（移行元TB vs kAIkei TB）
  → 仕訳件数検証
  → 貸借一致確認
  → [一致?]
    ─Yes→ 移行完了・ロールバックポイント削除
    ─No→ 差分分析 → 修正 or ロールバック

[Phase 5: 移行後確認]
  → ユーザー受け入れテスト
  → 本番運用開始
```

### 3.2 移行基準日の設定

| 項目 | 推奨 | 備考 |
|------|------|------|
| 移行基準日 | 月末（決済日） | 月次クローズと同時進行を推奨 |
| 移行期間 | 当期1月1日〜移行基準日 | 当期の全仕訳を移行 |
| 開始残高 | 前期末残高 | 移行基準日の期首残高として投入 |
| フリーズ期間 | 移行基準日の翌営業日〜移行完了 | 移行元ソフトへの新規入力を停止 |

## 5. バリデーションルール

### 5.1 移行前バリデーション（Dry-runフェーズ）

| ルールID | ルール名 | 条件 | エラーレベル |
|---------|---------|------|------------|
| MIG-VAL-001 | 科目コードマッピング存在 | 移行元科目コードがマッピング表に存在しない | エラー |
| MIG-VAL-002 | 補助科目マッピング存在 | 移行元補助科目がマッピング表に存在しない | 警告 |
| MIG-VAL-003 | 貸借一致 | 各伝票の借方合計=貸方合計 | エラー |
| MIG-VAL-004 | 金額範囲 | amount > 0 or amount < 0（0不可） | エラー |
| MIG-VAL-005 | 日付範囲 | transaction_dateが当期内 | 警告 |
| MIG-VAL-006 | 重複伝票番号 | 同一伝票番号が複数存在 | エラー |
| MIG-VAL-007 | 税区分マッピング存在 | 移行元税区分がマッピング表に存在しない | エラー |
| MIG-VAL-008 | 部門マッピング存在 | 移行元部門がマッピング表に存在しない | 警告 |
| MIG-VAL-009 | 開始残高貸借一致 | 開始残高仕訳の借方合計=貸方合計 | エラー |
| MIG-VAL-010 | 取引先マッピング存在 | 移行元取引先がマスタに存在しない | 警告（新規作成可） |

### 5.2 移行後検証

| 検証ID | 検証名 | 方法 | 合格基準 |
|--------|--------|------|---------|
| MIG-VRF-001 | 試算表一致 | 移行元TB vs kAIkei TB（全科目） | 全科目の残高が一致 |
| MIG-VRF-002 | 仕訳件数一致 | 移行元件数 vs kAIkei件数 | 差分0件 |
| MIG-VRF-003 | 貸借合計一致 | 全仕訳の借方合計=貸方合計 | 差額=0 |
| MIG-VRF-004 | 科目別残高一致 | 科目ごとの残高比較 | 全科目一致 |
| MIG-VRF-005 | 月次残高再計算 | monthly_balancesの自動再計算結果 | 計算結果=移行データ |
| MIG-VRF-006 | 固定資産残高 | 取得価額合計・累計償却額合計 | 移行元と一致 |

## 6. 差分レポート仕様

### 6.1 差分レポート出力項目

| 項目 | 説明 |
|------|------|
| 移行元伝票番号 | 移行元ソフトでの伝票番号 |
| 移行元取引日 | 移行元ソフトでの取引日 |
| 差分種別 | `missing`（移行先に存在しない）/ `mismatch`（金額・科目等の不一致）/ `extra`（移行先にのみ存在） |
| 移行元金額 | 移行元の借方/貸方金額 |
| 移行先金額 | kAIkeiの借方/貸方金額 |
| 差額 | 移行元金額 - 移行先金額 |
| 移行元科目 | 移行元の科目コード・名称 |
| 移行先科目 | kAIkeiの科目コード・名称 |
| 原因推定 | AIによる差分原因の推定（マッピング誤り・丸め差異・データ欠落等） |
| 推奨修正案 | 差分解消のための推奨アクション |

### 6.2 差分レポートフォーマット

```csv
migration_source_voucher,migration_source_date,diff_type,source_debit,source_credit,target_debit,target_credit,diff_amount,source_account,target_account,estimated_cause,recommended_action
Y-000123,2026/04/15,missing,50000,,0,,50000,1110 現金,,マッピング漏れ,科目マッピング表に1110を追加
Y-000124,2026/04/16,mismatch,10000,10000,10000,11000,-1000,4110 売上,4110 売上,消費税計算差異,税区分マッピングを確認（内税/外税）
```

## 7. ロールバック手順

### 7.1 ロールバック判定基準

| 条件 | ロールバック要否 |
|------|----------------|
| 試算表不一致の科目が3科目以内 | 修正で対応（ロールバック不要） |
| 試算表不一致の科目が4科目以上 | ロールバック推奨 |
| 仕訳件数の差分が1%超 | ロールバック推奨 |
| 貸借合計の不一致 | ロールバック必須 |
| システムエラーによる移行中断 | ロールバック必須 |

### 7.2 ロールバック手順

```
[1] 移行対象テナント・会社の特定
  → company_id / tenant_id を確認

[2] 移行データの特定
  → journal_headers where source_type = 'migration' AND company_id = ?
  → 移行日時以降に作成された全レコードを対象

[3] トランザクション開始
  BEGIN;

[4] 仕訳データの論理削除
  UPDATE journal_headers SET is_deleted = TRUE
    WHERE source_type = 'migration' AND company_id = ?;
  UPDATE journal_lines SET is_deleted = TRUE
    WHERE journal_header_id IN (上記のjournal_header_id);

[5] マスタデータの復元
  → 移行によって上書きされたマスタを移行前の状態に復元
  → バックアップスナップショットから復元（PITR）

[6] 月次残高の再計算
  → monthly_balances の移行対象月を再計算

[7] 検証
  → 移行前のTBと一致することを確認

[8] コミット
  COMMIT;

[9] ロールバック完了ログ記録
  → audit_trails に action=migration_rollback を記録
```

### 7.3 ロールバック後の対応

| 手順 | 内容 |
|------|------|
| 原因分析 | 差分レポート・エラーログから根本原因を特定 |
| マッピング修正 | 科目・税区分・部門マッピング表の修正 |
| 再シミュレーション | Dry-runを再実行してエラーが解消したことを確認 |
| 再移行 | 本番移行を再実行 |

## 8. 移行スケジュール例

| ステップ | 期間 | 作業内容 | 担当 |
|---------|------|---------|------|
| 事前準備 | 週1 | 移行元エクスポート・マッピング定義 | 移行チーム |
| Dry-run 1回目 | 週2 | テスト環境インポート・エラー確認 | 移行チーム |
| マッピング修正 | 週2 | エラー解消・マッピング修正 | 移行チーム |
| Dry-run 2回目 | 週3 | 再シミュレーション・検証 | 移行チーム |
| ユーザー確認 | 週3 | 差分レポートレビュー・承認 | 顧客 |
| 本番移行準備 | 週4 | 本番環境準備・フリーズ日程調整 | 移行チーム・顧客 |
| 本番移行 | 月末 | データエクスポート→インポート→検証 | 移行チーム |
| 移行後確認 | 翌週 | UAT・本番運用開始 | 顧客・移行チーム |

## 9. 移行時の注意事項

| # | 注意事項 | 対策 |
|---|---------|------|
| 1 | 移行元の科目体系がkAIkeiと異なる場合がある | 科目マッピング表を事前に作成し顧客に承認を得る |
| 2 | 移行元の税区分がkAIkeiの税区分に1:1で対応しない | 最も近い税区分にマッピングし、差分を警告表示 |
| 3 | 移行元の補助科目が階層構造を持つ場合がある | 階層を平坦化し、親子関係を別途記録 |
| 4 | 移行元の金額が整数のみの場合がある | kAIkeiはNUMERIC(15,4)のため、そのまま移行 |
| 5 | 移行元の証憑ファイルがローカル保存の場合がある | ファイルを一括エクスポートしてオブジェクトストレージへ移行 |
| 6 | 移行元の固定資産の償却方法がkAIkeiと異なる場合がある | 最も近い償却方法にマッピング |
| 7 | 大量データのインポートでタイムアウトが発生する可能性 | バッチ分割（1000件/バッチ）で段階的インポート |
| 8 | 移行元の仕訳に無効化（void）されたものが含まれる場合 | `is_voided=TRUE`として移行し、残高計算から除外 |
